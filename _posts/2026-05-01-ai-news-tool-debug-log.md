## 今回の作業はすべてCursor上で

前回の記事で触れた「現在地」も含め、今回の一連の作業はすべて **Cursor** 上で進めました。

| フェーズ | 使ったCursorのモード |
|---|---|
| ツール構造の理解・変更点整理 | Askモード |
| 変更プランの作成・確認 | Planモード |
| コーディング・リポジトリ作成 | Agentモード |

Agentモードでコーディングを始めると、変更プランの中で提案された「別リポジトリで進める」という方針に従い、AgentがGitHub上にプライベートリポジトリを自動で作成しました。

CursorからリモートのGitHubにリポジトリが生成される瞬間を目の前で見ていて、思わず「おー！」と声が出ました。

> コードは書いていない。でも、ちゃんと動くリポジトリが目の前に現れた。これがAgentモードの体感です。
{: .prompt-tip }

## 環境設定：APIキーの管理は1Password

コーディングの後、各種APIキーを取得しました。

10年以上使っている **1Password** でキーを管理し、GitHubのプライベートリポジトリの **Secrets設定** に登録。コード内にキーを直接書かない運用にしています。

> APIキーをコードに直接書くと、GitHubに上げた瞬間に流出します。Secretsに登録するのは最低限のセキュリティです。
{: .prompt-warning }

## 動作確認：Slackに届くまでの失敗2連発

コードには「ニュースソースが未指定でも稼働する初期設定」が組み込まれていたので、まずSlackに届くかだけを確認しました。

### 失敗① チャンネルIDではなくチャンネル名を指定していた

最初、Slackに何も届きませんでした。Cursor内でログを確認すると、Slackのチャンネル指定に **チャンネルID** ではなく **チャンネル名** を入れていたことが原因でした。

Slackは見た目の名前とシステム内のIDが別物です。IDに差し替えて、無事に届くことを確認。

### 失敗② GeminiのAPIが個人アカウントで上限に達していた

Slackには届いたものの、今度はコード内で設定されているエラー表示が出ました。調べると、**GeminiのAPIが無料枠の上限**に達していました。

個人アカウントで取得したAPIキーを諦め、**Google Workspaceアカウントで従量制のキーを再取得**。これで、エラーなくSlackにニュースが届くようになりました。

> Gemini APIの無料枠は思ったより早く消えます。テスト段階で使い切るケースが多いようです。
{: .prompt-info }

テストモードを本番モードに切り替え、設定した9つのニュースソースからSlackへの配信が確認できました。

## テスト中に気づいたこと

### GitHubとCursorを行き来する二度手間

GitHub Actionsのエラーを確認する際、最初は **ClaudeでエラーをチェックしてからCursorに戻る** という手順を何度か繰り返しました。

Cursorにまだ慣れていない部分が大きいですが、「どこで何を確認するか」の判断が定まっていないことが二度手間の原因でした。徐々に整理していきます。

### スクリーンショットがテキスト入力より圧倒的に速い

エラー内容を最初はキーボードで打ち込んでいましたが、途中で気づきました——**スクリーンショットを貼る方が速いし、抜け漏れがない**。

マコなりさんの講座で紹介されていた **Cleanshot** をすでに使っていたので、これをエラー確認に活用するようになりました。AIにとっては、テキストより画像の方が文脈ごと伝わります。

> エラーをAIに伝えるときはスクショが正解。文字起こしは抜け漏れが出やすく、AIの回答精度も下がります。
{: .prompt-tip }

### コードがわからなくても、Askモードで聞く習慣ができてきた

コードの一つ一つは今もわかりません。エラーが出ても「どこを見ればいいか」がわからない状態です。

それでも、**CursorのAskモードに切り替えて、わからないことをそのまま聞く**という習慣が、気づけばかなり自然にできるようになってきました。

「わかってから動く」ではなく、「動かしながら聞いて、少しずつわかる」サイクルに入ってきた感覚があります。

## 運用1日目：GitHub Actionsは「ぴったりの時刻」には動かない

今日から運用フェーズに入りました。コード内で **朝7時30分** に実行するよう設定していましたが、その時刻になっても何も動きませんでした。

調べると、6つほどの可能性が挙がりましたが、いちばん多い原因は——

> **GitHub Actionsのスケジュール実行は、指定時刻にぴったり動かないことがある。**  
> 特に混雑時間帯は、30分〜1時間以上ずれることが普通です。
{: .prompt-info }

対処は「しばらく待つ」。様子を見ていたところ、**1時間遅れで無事に実行されました。**

翌朝の実行履歴を確認するまで、実際のタイミングは確定しません。明日の朝、Actionsの「Scheduled」履歴を見てから、本当の意味で運用スタートと言えそうです。

---

💡 **OHOポイント:** 「コードがわからない」はもう言い訳にならない。Askモードとスクショがあれば、エラーのほとんどは前に進められる。
